# Protocol Tests

Future protocol validation should cover the fixed no-payload smoke trigger boundary, read-only status boundary, fixed `MoveAbsolute` boundary, fixed left-only `ClickAbsolute` boundary, and fixed single-key `KeyTap` boundary: unknown IOCTLs, trigger nonzero input/output buffers, status nonzero input or missing output buffer, exact command request size, version and command type checks, reserved-field rejection, out-of-range coordinate rejection, disconnected device, not-ready state, already-running state, already-completed smoke trigger state, and cleanup/deletion races.

Receipt/status validation should cover supported command mask, client diagnostic `SequenceId` echo behavior, nonzero driver receipt IDs for accepted commands, rejected-attempt fields, accept status for accepted receipts, release status, retry flags, and final-neutral-known fields. It should include the scenario where command A is active, command B is rejected, and command A then completes without mixing B identity into A receipt status. No raw HID payload API, queue, macro language, arbitrary button command, TypeText/string protocol, shortcut protocol, or repeat protocol exists. The current control surface is one fixed trigger for the built-in smoke sequence, one read-only fixed status query, one typed normalized absolute movement request, one typed normalized left-click request, and one typed single-key tap request using the driver-owned basic US HID keyboard mapping.
